1. PROJECT OVERVIEW
--------------------------------------------------------------------------------

Goal:
  Extend Frequency-Adaptive Dilated Convolution (FADC, CVPR 2024) from 2D
  natural image segmentation to 3D volumetric medical image segmentation,
  applied to breast tumor delineation in Dynamic Contrast-Enhanced MRI (DCE-MRI).

Why this matters:
  - Breast cancer diagnosis via DCE-MRI requires accurate tumor boundary
    delineation. Manual segmentation is time-consuming and expert-dependent.
  - Tumors vary enormously in size (2mm nodules to 80mm+ masses), making
    fixed-scale convolution inadequate.
  - Multi-center data (different scanners, protocols, populations) introduces
    domain shift — motivating federated learning in Phase 2.

Novel Contributions:
  1. First 3D extension of FADC for volumetric medical imaging.
  2. A new 3D reformulation of the FADC core mechanism that replaces the
     2D-only deformable convolution with parallel dilated Conv3d branches
     weighted by OmniAttention3D.
  3. Application to the large-scale multi-center MAMA-MIA dataset (1506 cases).
  4. Ablation study: FADC-Full vs FADC-Encoder vs FADC-Bottleneck.
  5. [Phase 2] Federated multi-center training across 4 breast MRI collections.

Target Publication:
  "Frequency-Adaptive Dilated Convolution for Federated Breast MRI Tumor
   Segmentation: A Multi-Center Study"
  Target Venues: Medical Image Analysis / IEEE TMI / MICCAI 2026

5A. MODEL ARCHITECTURE
-----------------------

All FADC variants use the unified UNet3DFADC class in models/unet_3d_fadc.py.
The fadc_placement parameter controls which blocks use FADCConvBlock vs standard
ConvBlock. Architecture depth and channel widths are identical across all
variants — only the conv block type changes.

  fadc_placement='full'       → FADC at encoder + bottleneck + decoder
  fadc_placement='encoder'    → FADC at encoder only
  fadc_placement='bottleneck' → FADC at bottleneck only

Key design decisions in the FADC modules (fadc_3d/):
  - k_list=[2] only (NOT [2,4,8]): prevents empty FFT masks at bottleneck depth=4
  - GroupNorm(1,C) in OmniAttention3D (NOT BatchNorm): prevents train/eval
    divergence when global avg pool reduces spatial to 1×1×1 with batch_size=2
  - dilation_list=[1,2,4]: 3 parallel branches, OmniAttention softmax-weights them
  - FrequencySelection3D zero-init: starts as identity, stable early training
  - lp_type='freq' (FFT-based): avgpool mode has shape bugs at stride=1

7. ENGINEERING ISSUES RESOLVED
--------------------------------------------------------------------------------

Issue 1 — Val Dice 0.03 (OmniAttention3D BatchNorm bug)
  BatchNorm after AdaptiveAvgPool3d produces noisy running stats with
  batch_size=2 (only 2 elements per channel). In eval() mode during
  validation, frozen bad stats → garbage attention weights → Dice 0.03
  while training Dice was 0.43. Fix: GroupNorm(1, attention_channel).
  GroupNorm is batch-size-independent, identical train/eval behavior.

Issue 2 — k_list=[2,4,8] empty FFT masks at bottleneck
  At 3D bottleneck depth=4, k=4 and k=8 produce empty cuboid masks.
  Empty mask → low_part=0 → high_part incorrectly equals previous residual.
  Fix: k_list=[2] only. Non-degenerate at all spatial sizes down to depth=4.

Issue 3 — OOM: FADC at 128×128×64
  3 parallel Conv3d branches tripled activation memory vs baseline.
  Fix: reduced FADC patch size to 96×96×48.

Issue 4 — Patch size not threading (OOM persisted despite --patch_size flag)
  PATCH_SIZE was a module-level constant in mama_mia_dataset.py, independent
  of CLI. Fix: renamed to DEFAULT_PATCH_SIZE, threaded patch_size through
  get_train_transforms, get_rand_train_transforms, PreprocessedDataset,
  build_centralized_loaders.

Issue 5 — Kaggle .npz cache corruption
  Original upload corrupted many files. Fix: re-compressed clean local cache,
  re-uploaded using kaggle==1.6.17 (v2.x had broken auth).

Issue 6 — Validation too slow (1.5h per run with FADC)
  FADC's FFT operations made each forward pass ~3x slower than baseline.
  With overlap=0.25 and 306 full volumes, validation took 1.5 hours.
  5 validations in 50 epochs = 7.5h → exceeded Kaggle 12h session limit.
  Fix: overlap=0.25 → overlap=0.0 (~35 min per validation).
       val_every=10 → val_every=25 (2 validations per run).

Issue 7 — val_every change didn't take effect immediately
  Validation still fired at epoch 20 because the running process used
  the config from the checkpoint session (val_every=10). Not a bug —
  the fix is active from epoch 21 onwards.

Issue 8 — Sigmoid gating signal suppression (~75% loss)
  AdaptiveDilatedConv3D.forward() multiplied features by c_att (sigmoid)
  and f_att (sigmoid) without the *2 compensation used in the original
  2D FADC code. sigmoid avg ≈ 0.5, so 0.5 × 0.5 = 0.25 signal retained.
  Fix: c_att * 2, f_att * 2 (sigmoid(0.5)→1.0 = identity at average).

Issue 9 — Missing residual connections in FADCConvBlock
  Original 2D FADC was designed for ResNet/FPN (built-in residuals).
  Our U-Net ConvBlocks had no skip path, forcing all gradients through
  FreqSelect → OmniAtt → 3 branches → gating — poor gradient flow.
  Fix: Added 1x1 Conv3d projection skip in FADCConvBlock.

Issue 10 — No LR warmup for FADC modules
  FADC attention starts random; full 1e-4 LR from epoch 1 pushes
  random gates into bad states. Original uses ImageNet-pretrained features.
  Fix: 5-epoch LinearLR warmup (start_factor=0.1) before cosine annealing.
